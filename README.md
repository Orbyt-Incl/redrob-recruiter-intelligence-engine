# Redrob — Intelligent Candidate Discovery & Ranking

Team **Orbyt** · Redrob Hackathon · *Senior AI Engineer — Founding Team* JD

Ranks the top 100 of 100,000 candidates by reasoning about **inferred capabilities and
hireability**, not keyword overlap. The system is built to surface "hidden gems" (people who
*built* recommendation / ranking / retrieval systems at product companies without listing the
buzzwords) and to resist the dataset's traps (keyword-stuffers, consulting-only careers,
job-hoppers, CV/speech-only profiles, and the ~80 impossible "honeypot" profiles).

## How it works

Two phases. **Precompute** is offline and unlimited; **ranking** is CPU-only, offline, < 5 min.

```
precompute.py  ──>  artifacts/                    rank.py  ──>  submission.csv
  jd_requirements.json                              FAISS shortlist (100k -> 1500)
  features.parquet  (+ honeypot flags)              Recruiter Intelligence Score
  capability_scores.parquet                         honeypot hard-filter
  candidate_embeddings.faiss                         top-100 + factual reasoning
  id_index.parquet, jd_embedding.npy                 validate against the spec
  anchor_embeddings.npy
  embedding_model/  (local; no network at rank)
```

**Key idea — capability inference from career evidence.** Capability keywords (retrieval,
ranking, recommendation, embeddings, evaluation, production ML, …) are matched against
*career-history descriptions* — what the candidate actually did — **not** the skills list, which
stuffer traps pack with AI tokens. A `CAPABILITY_ANCHORS` semantic layer (CPU sentence-transformer)
adds keyword-free credit. See `src/capability_graph.py`.

**Recruiter Intelligence Score** (`src/scoring.py`) blends capability fit, career evidence
(product-company credit *gated* on real capability), availability (behavioral signals), recruiter
demand, and experience fit, minus soft penalties (job-hopping, consulting-only, research-only,
CV/speech-only, title mismatch, location/notice). Honeypots (`src/honeypot.py`) are the **only**
hard filter.

pip install -r requirements.txt

Tested with **Python 3.10.11** on a 16 GB CPU-only machine (see `submission_metadata.yaml`).

## Reproduce the submission

The dataset file `candidates.jsonl` (~465 MB, 100k profiles) is provided in the hackathon bundle
and is **git-ignored**; drop it in the repo root before running.

```bash
# Phase 1 — build artifacts (downloads the embedding model ONCE, then fully offline; ~15 min)
python precompute.py --candidates ./candidates.jsonl

# Phase 2 — produce the submission (CPU only, no network, < 5 min)
python rank.py --candidates ./candidates.jsonl --out ./orbyt.csv --validate
```

**Stage-3 single command.** The committed `artifacts/` are produced by `precompute.py`, so the
ranking step reproduces the CSV on its own — no re-precompute required:

```bash
python rank.py --candidates ./candidates.jsonl --out ./orbyt.csv
```

`rank.py` reads only the precomputed artifacts (keyed by `candidate_id`) plus the local FAISS
index; it never loads the embedding model and makes **no** network calls. `--candidates` is
accepted for interface parity but the ranking step consumes the artifacts, not the raw file.

## Evaluate (proxy — no public ground truth)

```bash
python -m eval.proxy_eval --submission orbyt.csv
```

Reports proxy NDCG@10/50, MAP, P@10 and runs **trap regression checks** (0 honeypots in top-100,
keyword-stuffers suppressed, product-experience dominance, hidden gems surfaced). The labels are a
heuristic proxy, **not** ground truth — used only for local sanity/tuning.

## Notes

- **AI tool use:** Claude (Claude Code) was used for design and authoring. No candidate data is
  sent to any hosted LLM at rank time. The optional `src/enrich_llm.py` enrichment runs offline in
  precompute and is disabled by default.
- **Compute constraints honored:** ranking is CPU-only, single machine, no GPU, no network,
  well under the 5-minute / 16 GB / 5 GB-disk limits.
- `candidates.jsonl` and `artifacts/embedding_model/` are git-ignored (size); regenerate with
  `precompute.py`. The FAISS index is tracked via Git LFS (see `.gitattributes`).
